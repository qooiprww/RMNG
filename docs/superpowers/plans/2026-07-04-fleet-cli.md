# Fleet CLI (`rmng`) + MCP slimming

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:executing-plans (or
> subagent-driven-development). Tasks are ordered; each lands with its tests.

**Goal:** Replace MCP-based fleet management with a first-class CLI. A new
`crates/cli` binary (`rmng`) drives the port-2 web API; the port-4 (:9003) MCP
keeps ONLY the proxied desktop/computer-use tools; the per-clone (:9002) MCP is
untouched. The CLI is injected into every clone at `/usr/local/bin/rmng` so
agents inside sandboxes manage the fleet with it (`RMNG_CONTROL_URL` is already
preset in every clone).

**Why:** agent fleets are orchestrated from CLIs (composable, scriptable,
JSON-outputtable), not MCP tool calls. Desktop automation stays MCP because
screenshots-as-tool-results are exactly what MCP is good at.

**Architecture:** `wire` types are the single source of truth for shapes;
`control-client` becomes the typed HTTP+SSE client the CLI links; no new server
subsystems — only two small web additions (`GET /api/state`, a `hostname` mode
on `POST /api/clone`) restoring parity with what the removed MCP tools could do.

## Global constraints

- No new server capabilities beyond `GET /api/state` + clone `hostname` mode:
  no exec, no file copy (workers are driven via computer use; data moves via git).
- `--json` output is the wire type serialized verbatim — never a bespoke shape.
- `rmng wait` rides `/events` SSE (ops are pruned 8s after Done / 60s after
  Error — polling races the prune).
- Per-clone `set_state` (:9002) must keep working byte-identically (the
  agent-wrapper depends on it).
- Conventional commits; each task's tests land in the same commit.

## Verification

- `cargo test -p wire -p control-client -p rmng-cli -p control-server`
- Isolated integration (docker-in-docker in a clone): dev control-server,
  `rmng ps` (empty) → `image pull --wait` → `clone --hostname t1` → `ops` → `rm`.
- Live read-only smoke against `http://rmng-control:9000`: `ps`, `image ls`,
  `account ls`, `ops` (SSE fallback until the new server deploys).
- `:9003 tools/list` shows only desktop tools; `:9002` still serves `set_state`.

## Risks

- Anything driving fleet tools on :9003 breaks at deploy (not merge) — release
  note it; chat stays on `POST /api/chat/:id`, everything else moves to the CLI.
- Existing clones don't get the CLI (create-time injection is the only delivery
  path since the binswap engine was retired) — recreate or install manually.
- No auth on port 2: the CLI in every clone lowers the accident bar for
  fleet-wide `rm` — mitigated with an interactive confirm (`--yes` to skip);
  a real authz story is out of scope here.
